# HSL Transport RAG Assistant

A Retrieval-Augmented Generation (RAG) agent designed to answer localised transportation queries, city bike capacities, and transit terms for HSL (Helsinki Regional Transport Authority) in Finland.

Two parallel implementations are provided — a linear LangChain pipeline and a LangGraph agent — both sharing a single offline FAISS knowledge base.

---

## Repository Structure

```
HSL-TRANSPORT-ASSISTANT-RAG/
├── langChain/
│   └── app.py               # Linear RAG pipeline with Panel UI dashboard
├── langGraph/
│   └── agent.py             # Essay Writer-style LangGraph agent
├── ingest.py                # Shared data ingestion — run once, used by both
├── faiss_hsl_index/         # Generated by ingest.py
├── hsl_policy_terms.docx    # Local HSL policy document — required by ingest.py
├── .gitignore
└── README.md
```

## Step 1 — Install Dependencies

### LangChain implementation

```bash
pip install requests panel langchain langchain-community langchain-huggingface \
            langchain-ollama faiss-cpu docx2txt python-dotenv
```

### LangGraph implementation

```bash
pip install langgraph langchain-core langchain-ollama langchain-community \
            langchain-huggingface faiss-cpu pydantic python-dotenv
```

> You can install both sets together if you want to run either implementation from the same environment.

---

## Step 2 — Configure API Key

Open ingest.py and paste your digitransit key inside the DIGITRANSIT_KEY string variable.

---

## Step 3 — Run Ingestion (run once, shared by both)

`ingest.py` lives at the repo root and is shared by both implementations.
It connects to the Digitransit GraphQL API, fetches routes, bike station data,
and service alerts, reads `hsl_policy_terms.docx`, and builds the FAISS index.

Make sure `hsl_policy_terms.docx` is saved at the repo root, then run:

```bash
# from the repo root
python3 ingest.py
```

This creates the `faiss_hsl_index/` folder at the repo root.
You never need to run this again unless you change or add source documents.

---

## Step 4 — Run an Implementation

### LangChain — Panel UI dashboard

Launches a browser-based interactive dashboard at `http://localhost:5006/app`.

```bash
cd langChain
python3 -m panel serve app.py --show
```

### LangGraph — terminal agent

Runs a single question through the LangGraph agent pipeline and prints each node's output.
Change the question at the bottom of `agent.py` before running.

```bash
cd langGraph
python3 agent.py
```

To change the question, edit the last lines of `langGraph/agent.py`:

```python
if __name__ == "__main__":
    run(
        task="Which trams stop at Pasila?",
        max_revisions=2,
    )
```

## How the LangGraph Agent Works

The agent follows the below pattern.

```
planner → research_plan → generate → reflect
                               ↑          |
                               |    NO GROUNDING FAILURE → END
                               |
                         research_critique
```

| Node | Role |
|---|---|
| `planner` | Summarises the user query into a compact search-intent string |
| `research_plan` | Generates keyword queries, retrieves FAISS chunks, filters with a condenser LLM |
| `generate` | Writes or revises the answer draft from filtered context |
| `reflect` | Evaluates the draft for grounding failures against the retrieved chunks |
| `research_critique` | Re-retrieves from FAISS based on the critique's identified gaps |

The loop exits when the reflection node outputs `NO GROUNDING FAILURE` or `max_revisions` is reached.

## Notes

- `faiss_hsl_index/` is listed in `.gitignore` and must be generated locally — it is not committed to the repository.
- `hsl_policy_terms.docx` is also excluded from version control. Place it at the repo root before running ingestion.
- Both implementations are fully offline after the index is built — no internet connection required at query time.
